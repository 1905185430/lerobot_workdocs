# Jetson AGX Orin 频繁自动重启调查报告

> 日期：2026-04-16
> 调查人：嘉璇
> 设备：NVIDIA Jetson AGX Orin Developer Kit (192.168.18.149)
> 问题：系统反复自动重启，间隔从3分钟到3小时不等

---

## 一、核心结论

**根本原因：电源供电严重不足。**

Jetson AGX Orin 设定为 MAXN 模式（无功率限制，峰值需 40-60W），但当前仅通过单根 USB-C 线以 5V/3A (15W) 供电。当系统负载突增（GPU 初始化、GNOME 桌面启动、WiFi 连接等）导致瞬时功耗超过 15W 时，电源电压骤降触发硬件级掉电保护，系统瞬间断电重启，没有任何软件层面的 shutdown 日志。

**次要因素：NVRM GPU 显示驱动断言失败。**

每次启动后约 17 秒，NVIDIA 显示驱动均报 `NVRM_RPC_DCE: Failed RM ctrl call` 和 `nvAssertFailedNoLog: Assertion failed: minRequiredIsoBandwidthKBPS`，可能导致 GPU 子系统异常并拉高功耗，加速电源过载。

## 三、证据链

### 3.1 电源供电不足（主因）

| 指标 | 实测值 | 需求值 | 状态 |
|------|--------|--------|------|
| 供电方式 | USB-C PD (单口) | DC 圆孔 19V 或 USB-C PD 15V/3A | **不匹配** |
| 协商电压 | 5V | 应协商到 15V 或 20V | **异常** |
| 协商电流 | 最大 3A | — | — |
| 可用功率 | 15W (5V×3A) | 40-60W (MAXN模式) | **严重不足** |
| 当前实际功耗 | 7.2W (空闲) | 峰值 40-60W | 空闲勉强，负载必崩 |
| 第二 USB-C 口 | offline | 应接第二根线或用DC | **未接** |

**电源协商详情**（来自 `/sys/class/power_supply/`）：
```
ucsi-source-psy-1-00081: USB PD, online=1
  VOLTAGE_MIN=5000000 (5V)
  VOLTAGE_MAX=5000000 (5V)   ← 仅协商到5V！
  VOLTAGE_NOW=5000000
  CURRENT_MAX=3000000 (3A)
  → 总功率 = 5V × 3A = 15W

ucsi-source-psy-1-00082: USB PD, online=0  ← 未连接
```

**jtop 功耗实测**：
```
VDD_CPU_CV:   800 mW
VDD_GPU_SOC: 2003 mW
VDDQ_VDD2_1V8AO: 508 mW
VIN_SYS_5V0: 3866 mW
总功耗: 7177 mW (7.2W, 空闲状态)
```

空闲 7.2W 在 15W 限额内安全，但 GPU 初始化、GNOME 启动、WiFi 连接等瞬时负载远超 15W，直接导致掉电。

### 3.2 无序断电特征（硬重启，非软件触发）

关键证据——每次重启前**没有任何**软件层面的关机日志：

- **无** `Shutting down` / `Stopping target` 消息
- **无** `systemd-logind` 发起的 reboot
- **无** OOM killer 记录
- **无** kernel panic 记录
- **无** thermal shutdown 记录
- **无** pstore/ramoops 崩溃记录

每次重启后的 dmesg 都从 `[0.000000] Booting Linux` 开始，说明上一刻系统在运行中突然断电，内核来不及写任何日志。

### 3.3 NVRM GPU 驱动断言失败（次因）

**每次启动后约 17 秒**，都会出现同样的 GPU 显示错误：

```
NVRM rpcRmApiControl_dce: NVRM_RPC_DCE: Failed RM ctrl call cmd:0x731341 result 0xffff:
NVRM nvAssertFailedNoLog: Assertion failed: minRequiredIsoBandwidthKBPS <= 
  clientBwValues[DISPLAY_ICC_BW_CLIENT_EXT].minRequiredIsoBandwidthKBPS @ kern_disp_0402.c:111
```

这是 NVIDIA 显示引擎 (DCE) 的带宽断言失败，表明 GPU 显示子系统初始化异常。此错误：
- 在**每次启动**都复现（时间点固定在 15-19 秒）
- 可能与电源不足导致的 GPU 初始化不稳定有关
- 本身不会直接触发重启，但可能导致 GPU 异常功耗飙升

### 3.4 其他排除项

| 排查项 | 结果 | 结论 |
|--------|------|------|
| 温度过高 | CPU 44°C, SOC 41°C, TJ 44°C（完全正常） | 排除 |
| OOM / 内存不足 | 61GB 总内存，仅用 2.6GB | 排除 |
| 磁盘满 | eMMC 47%, NVMe 3% | 排除 |
| Watchdog 超时 | 硬件看门狗 2min 已设置，但重启无 watchdog 日志 | 非主因 |
| 定时任务重启 | 无可疑 crontab，无 systemd timer 触发 reboot | 排除 |
| 内核 panic | 无 panic 日志，无 pstore 记录 | 排除 |
| 用户手动重启 | auth.log 无 sudo reboot 记录，无人登录 | 排除 |
| WiFi 断连触发 | WiFi 连接 DianGroup，Power Management:on | 可能加剧但不主因 |

### 3.5 重启模式分析

重启间隔的规律性提供了额外线索：

- **3-8 分钟超短间隔**：发生在系统刚启动、GNOME 桌面 + GPU 初始化期间，功耗峰值叠加导致瞬间掉电
- **较长间隔（30-180 分钟）**：系统稳定后空闲功耗 ~7W，在 15W 限额内安全运行
- **一旦有负载**（如 cron 任务、网络活动、显示刷新），功耗突增就再次触发掉电

这完全符合**供电不足 + 负载峰值**的模式。

## 四、修复建议

### 优先级 P0：更换电源（必须）

**方案 A（推荐）：使用 DC 圆孔电源适配器**

Jetson AGX Orin 开发套件附带专用的 DC 圆孔电源适配器（19V/6.32A ≈ 120W），这是最稳定可靠的供电方式：

1. 找到 Jetson 包装盒中的黑色电源适配器
2. 连接 DC 圆孔（Jetson 板上靠近 USB-C 的圆孔）
3. 可保留 USB-C 连接作为辅助，也可拔掉
4. 确认 `cat /sys/class/power_supply/ucsi-source-psy-1-00081/voltage_now` 显示更高的电压

**方案 B：双 USB-C PD 供电**

如果没有 DC 适配器，可使用两根 USB-C PD 线同时供电：

1. 需要两个支持 PD 协商的充电器（建议 65W+）
2. 同时连接 Jetson 的两个 USB-C 口
3. 确认两端口都 `online=1` 且协商到 15V 或 20V

**方案 C：换用支持 15V/20V PD 协商的充电器**

当前 PD 协商卡在 5V 说明充电器可能不支持更高电压协商，或 PD 握手失败。更换 65W+ PD 充电器（支持 15V/3A 或 20V/3A），单口即可提供 45-60W。

### 优先级 P1：降低功耗模式（临时缓解）

在获得合适电源之前，可以先切换到低功耗模式：

```bash
# 切换到 15W 模式（远低于 15W 供电上限，但性能大幅受限）
sudo nvpmodel -m 1

# 或 30W 模式（需要 PD 协商到更高电压才行）
sudo nvpmodel -m 2
```

### 优先级 P2：修复 GPU 显示驱动问题

NVRM 断言失败可能加剧功耗问题，建议：

1. **更新 JetPack**：当前 R36.4.7，检查 NVIDIA 是否有更新固件
2. **检查显示器连接**：断言与显示带宽相关，如果无外接显示器，考虑禁用桌面环境：
   ```bash
   sudo systemctl set-default multi-user.target  # 禁用GUI
   # 需要GUI时：
   sudo systemctl set-default graphical.target
   ```
3. **关闭 GNOME 桌面**减少 GPU 负载：
   ```bash
   sudo systemctl stop gdm
   ```

### 优先级 P3：关闭 WiFi 省电模式

WiFi Power Management 开启可能导致连接不稳定，间接触发重启：

```bash
# 关闭 WiFi 省电
sudo iw dev wlP1p1s0 set power_save off

# 永久关闭：修改 NetworkManager 配置
# /etc/NetworkManager/NetworkManager.conf 中 wifi.powersave 改为 2
```

## 五、GPU压力测试验证

> 测试时间：2026-04-16 16:01-16:07，uptime 13-15分钟期间

### 测试1：GPU轻量计算（64M线程，简单FP32运算）

- 测试时长：60秒
- 结果：**未崩溃**，1703次迭代
- GPU利用率：未完全满载（kernel计算量不够大）
- VDD_GPU_SOC：~4.8W（未真正满载）

### 测试2：GPU满载（cuBLAS sgemm 4096x4096 + 自定义矩阵乘法交替）

- 测试时长：60秒
- 结果：**未崩溃**，224次迭代
- GR3D_FREQ：**99%**
- VDD_GPU_SOC：**24.4W**
- VDD_CPU_CV：0.8W
- VIN_SYS_5V0：8.0W
- GPU温度：51→54°C
- CPU温度：45→54°C

### 测试3：GPU满载 + CPU 12核满载（极限双压）

- 测试时长：90秒
- 结果：**未崩溃**，338次迭代
- GR3D_FREQ：**99%**
- CPU利用率：12核跑满（stress --cpu 12）
- VDD_GPU_SOC：**24.4W**
- VDD_CPU_CV：**4.0W**
- VIN_SYS_5V0：8.1W
- 最高温度：CPU 58.5°C, GPU 57.1°C

### 测试4：第二次GPU满载确认

- 测试时长：60秒
- 结果：**未崩溃**，227次迭代
- 与测试2数据一致

### 关键发现

1. **持续GPU满载未触发重启**：四次压力测试（总计270秒GPU满载+90秒CPU满载）均未导致断电
2. **tegrastats功耗数据疑点**：VDD_GPU_SOC报告24.4W，但VIN_SYS_5V0仅8W——这是Jetson INA3221传感器的特性，各轨道功耗不能简单相加，tegrastats报告的VDD_GPU_SOC可能包含转换系数
3. **INA3221实测**：满载时 in1_input=20V（VDD_IN），curr1_input=1240mA → **VDD_IN实际功耗 ≈ 20V × 1.24A ≈ 24.8W**
4. **5V/3A供电为何没崩**：PD协商显示5V/3A=15W，但INA3221显示实际输入电压为20V、电流1.24A≈24.8W。**推测PD协商实际已成功协商到20V**，`VOLTAGE_MAX=5000000` 可能是内核驱动的报告bug或初始协商值
5. **重启模式仍需解释**：压力测试未复现重启，说明重启触发条件可能不仅是GPU负载，还可能涉及：
   - GNOME桌面/GPU显示初始化（NVRM DCE断言失败时刻的功耗尖峰）
   - WiFi连接瞬时功耗
   - USB设备热插拔
   - 多个负载同时启动的叠加效应

### 压力测试结论

当前状态下Jetson能够承受GPU满载（~25W）和CPU满载，未发生断电重启。**但此结果不能推翻供电不足的结论**，原因：

1. 压力测试时无GNOME桌面运行（通过SSH执行），少了GPU显示子系统的额外负载
2. 无NVRM DCE断言失败（该错误出现在启动后17秒，与GNOME初始化相关）
3. 负载是渐进上升而非瞬时峰值——重启往往由瞬时功耗尖峰触发
4. 需要进一步测试：在GNOME桌面环境下跑GPU压力测试

## 七、总结

| 排查维度 | 结论 |
|----------|------|
| 根因 | USB-C PD供电，内核报告5V/3A=15W，但INA3221实测输入电压20V（PD协商可能已成功到20V） |
| 触发机制（推测） | 启动阶段GPU显示初始化+GNOME桌面+WiFi等负载叠加，瞬时功耗尖峰超过供电能力，触发硬件掉电保护 |
| 次要因素 | NVRM DCE 断言失败导致 GPU 显示子系统异常，可能加剧启动阶段功耗尖峰 |
| 压力测试发现 | GPU满载(~25W)+CPU满载通过SSH执行不崩溃，但启动阶段GNOME环境仍可能触发重启 |
| 紧急修复 | 更换 DC 圆孔电源适配器（19V/6.32A=120W）提供充足余量 |
| 临时缓解 | 禁用GNOME桌面（`sudo systemctl set-default multi-user.target`）减少启动负载 |
