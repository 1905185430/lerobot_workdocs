## 二、硬件调试

### 2.1 SO-101 机械臂环境搭建

- 安装了 Seeed 维护版 LeRobot（fork v0.4.3, Python 3.10）和官方 LeRobot（v0.5.1, Python 3.12）两套环境并存
- 完成从臂/主臂的首次标定，生成校准文件

### 2.2 波特率问题（关键 bug）

| 波特率 | 9600 | ... | 500000 | **1000000** |
|--------|------|-----|--------|-------------|
| 结果   | OK   | OK  | OK     | **FAILED**  |

**根因**：USB 转串口芯片 CH9101F 在 1Mbps 下通信不稳定，而 LeRobot 默认波特率恰好是 1000000。

**修复**：修改 `feetech.py` 中 `DEFAULT_BAUDRATE = 500_000`，两套环境都需改，且 `git pull` 可能覆盖。

### 2.3 摄像头识别

本机与 Jetson 摄像头设备号完全不同：

| 摄像头 | 本机 | Jetson | 格式 |
|--------|------|--------|------|
| 俯视（camera1） | video2 | video0 | MJPG |
| 腕部（camera2） | video10 | video6 | MJPG |

> 必须在每台机器上实测确认，不可直接复用配置。

## 三、技术调研

共完成 6 份调研文档，核心结论：

### 3.1 Seeed Fork vs 官方 LeRobot 版本兼容性

| | Seeed Fork | 官方最新 |
|---|---|---|
| 版本 | 0.4.4 | 0.5.1+ |
| 落后官方 | 143 commits | — |
| 数据集格式 | CODEBASE_VERSION v3.0（一致） | v3.0 |

**结论**：Parquet 数据文件格式完全一致，但 info.json 元数据有变化（codebase_version、fps、next.done 等字段），需做版本适配。

### 3.2 数据集版本兼容性（0.4.x → 0.5.x）

```
0.4.4 采集的 info.json          0.5.2 训练期望的 info.json
codebase_version: "v3.0"   →    codebase_version: "v0.5.2"
视频字段: "info"            →    "video_info" + fps
无 next.done                →    必须有
有 data_files_size_in_mb    →    需删除
```

**解决方案**：在 HF 上修改 info.json 后打 revision tag（v0.5.2），训练时指定 `--dataset.revision=v0.5.2`。

### 3.3 SmolVLA 小数据集微调效果分析

论文数据（50 episodes/任务，SO100 真实臂）：

| 策略 | 参数量 | 预训练 | 平均成功率 |
|------|--------|--------|-----------|
| ACT（从头训练） | ~10M | 无 | 48.3% |
| SmolVLA（无预训练） | 0.45B | 无 | 51.7% |
| SmolVLA（有预训练） | 0.45B | 481 数据集 | **78.3%** |

**结论**：预训练是最关键因素。我们用 30 episodes 微调有预训练的 SmolVLA，预期效果优于同数据量的 ACT。

### 3.4 Jetson AGX Orin 部署可行性

- 61.4GB 统一内存，可加载 SmolVLA（~3GB）
- PyTorch 2.5.0 Jetson 定制版，支持 FP16 推理
- 三方案对比：直接推理 > 模型量化 > 重训小模型
- **主要障碍**：无法直连 huggingface.co，需 HF_ENDPOINT=https://hf-mirror.com

## 四、模型训练

### 4.1 训练环境

- **平台**：PPIO GPU 云服务器（美国加州，RTX 4090 24GB）
- **数据集**：Ready321/vla_grab_redcube（30 episodes, 12714 frames, SO-101 抓红方块, 双摄像头）
- **模型**：SmolVLA (lerobot/smolvla_base → Ready321/my_smolvla)

### 4.2 训练配置

```bash
lerobot-train \
  --policy.path=lerobot/smolvla_base \
  --policy.repo_id=Ready321/my_smolvla \
  --dataset.repo_id=Ready321/vla_grab_redcube \
  --dataset.revision=v0.5.2 \
  --batch_size=64 \
  --steps=10000 \
  --save_freq=2000 \
  --rename_map='{"observation.images.top": "observation.images.camera1",
                 "observation.images.wrist": "observation.images.camera2"}' \
  --policy.empty_cameras=1
```

关键决策：
- 步数从 20K 降到 10K（30 episodes 数据量少，过多会过拟合）
- `empty_cameras=1`：只有 2 个摄像头，SmolVLA 期望 3 个，第 3 个空帧填充
- `rename_map`：数据集摄像头名 top/wrist 映射到模型期望的 camera1/camera2

### 4.3 训练结果

- Loss 从 ~0.3 下降到 ~0.008
- Step 6000 checkpoint 已上传到 HuggingFace: `Ready321/my_smolvla`
- 模型大小：~900MB（action expert）+ ~2GB（VLM 主干 SmolVLM2-500M）

## 五、本地部署与推理评估

### 5.1 部署过程

在 RTX 4060 Laptop (8GB) 上完成部署，GPU 占用约 90%（7.2GB）。

### 5.2 关键兼容性修复（共 5 个）

| # | 问题 | 根因 | 修复 |
|---|------|------|------|
| 1 | **SmolVLA 未注册** | 循环导入导致注册装饰器未执行 | 在 lerobot_record.py 加 import SmolVLAConfig |
| 2 | config.json 含 use_peft | 0.5.x 写入，0.4.4 不认识 | 删除该字段 |
| 3 | 摄像头名不匹配 | front/side ≠ camera1/camera2 | 直接用 camera1/camera2 命名 |
| 4 | policy.path 含 ~ | 未展开被当 repo_id | 用绝对路径 |
| 5 | 缺 num2words | SmolVLM processor 依赖 | pip install |

**其中最关键的是第 1 个**：SmolVLA 未注册导致 policy 加载静默失败，机械臂空转，日志仅打印 "No policy or teleoperator provided"，没有任何报错提示，排查耗时最长。

### 5.3 最终评估命令

```bash
HF_ENDPOINT=https://hf-mirror.com lerobot-record \
  --robot.type=so101_follower \
  --robot.port=/dev/ttyACM0 \
  --robot.id=so101_cong_left \
  --robot.cameras='{ camera1: {type: opencv, index_or_path: 2, width: 640, height: 480, fps: 30},
                     camera2: {type: opencv, index_or_path: 10, width: 640, height: 480, fps: 30} }' \
  --dataset.single_task="Grab the red cube to the white cup" \
  --dataset.repo_id=Ready321/eval_grab_redcube_test \
  --dataset.episode_time_s=50 \
  --dataset.num_episodes=10 \
  --dataset.push_to_hub=false \
  --policy.path=/home/xuan/下载/my_smolvla \
  --policy.empty_cameras=1 \
  --teleop.type=so101_leader \
  --teleop.port=/dev/ttyACM1 \
  --teleop.id=so101_zhu_left
```

**结果**：机械臂在 policy 驱动下自主执行抓取动作，评估进行中。

## 八、关键经验总结

1. **硬件问题先排查底层**：波特率、串口权限、摄像头设备号，看似软件问题实际是硬件/驱动兼容性
2. **版本兼容性是最大隐患**：LeRobot 0.4.x ↔ 0.5.x 之间 info.json 格式变化、config 字段增减，需要逐个适配
3. **静默失败最难排查**：SmolVLA 注册 bug 导致 policy 加载失败但无报错，只表现为机械臂空转
4. **每台机器硬件不同**：摄像头设备号、格式支持、串口映射都需要实测，配置不可直接复用
5. **国内网络需镜像**：huggingface.co 访问不稳定，HF_ENDPOINT=https://hf-mirror.com 是必备配置
